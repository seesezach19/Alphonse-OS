# Alphonse V0.2 Debug Loop Product Direction

## Objective

Alphonse is the provider-neutral governance and debugging environment for Agent Workflows. V0.2 proves one complete loop: connect an existing automation, report a demonstrated failure, reproduce it, commission a replaceable repair worker, independently verify the candidate, and promote it through explicit human authority.

The narrow product uses permanent architecture. It does not attempt to certify overall agent quality or support every runtime before proving the repair lifecycle.

## Product Horizons

1. Debug Loop: reactive diagnosis, reproduction, repair, verification, and target-native delivery.
2. Governed Development Environment: multiple workflows, runtimes, environments, CI, history, and promotion.
3. Agentic Operating Layer: live authority, effects, context, accountability, recovery, Butler, and portable Operational Packages.

Long-term Alphonse is batteries-included and generic: rich shared primitives, an opinionated lifecycle, replaceable intelligence, and pluggable runtime, repair, model, repository, data, and deployment edges.

## First External Substrate

V0.2 uses a separately operated customer-owned n8n instance. n8n owns workflow execution, provider integrations, and provider credentials. Alphonse owns workflow identity, revision attribution, diagnostic cases, repair orchestration, verification evidence, and promotion authority.

The first workflow compares ERP and storefront inventory, classifies fulfillment risk, drafts a customer follow-up with a model, and routes the draft only to local review. A seeded mapping defect converts a missing ERP SKU into zero inventory, producing a successful but false delay email. Correct behavior preserves `inventory_unknown` and routes human review.

## V0.2 Journey

1. Install customer-controlled n8n beside Alphonse Node.
2. Install and configure the trusted first-party n8n Operational Package.
3. Import the Alphonse Event Reporter subworkflow.
4. Receive a signed minimal Runtime Event Envelope for an execution.
5. Resolve exact workflow JSON, n8n image, node, model, and configuration revision.
6. Builder reports the false inventory email outcome.
7. Create a Diagnostic Case and human-confirmed Failure Specification.
8. Pull only the detailed execution data needed for investigation.
9. Produce a redacted immutable Reproduction Bundle with deterministic external fixtures.
10. Attach a customer-controlled Repair Worker under a Passport and lease.
11. Produce an inactive candidate n8n workflow and targeted regression artifact.
12. Run original and candidate through an independent deterministic Verification Runner.
13. Emit a signed Verification Receipt.
14. Require explicit human promotion through the n8n Repair Delivery Adapter.
15. Confirm the promoted target revision and retain rollback reference.
16. Preserve the demonstrated failure as a reusable targeted regression.
17. Repeat with a model-assisted Diagnostic Worker after the deterministic path passes.

## Architecture

### Alphonse Node

One customer-controlled Docker deployment contains Kernel authority services, Diagnostic Plane metadata and artifacts, Agent Gateway, diagnostic orchestration, worker coordination, Verification Runner, CLI, and later Builder Console. One Node serves one customer trust boundary and may contain many workflows and separately authoritative environments.

### Stable Product Objects

- Agent Workflow: durable operational objective across implementations.
- Agent Revision: exact available code, workflow, model, tool, and policy identity.
- External Activity Trace: untrusted observation outside Kernel admission.
- Runtime Event Envelope: signed provider-neutral external lifecycle claim.
- Diagnostic Case: builder investigation spanning exact observations and revisions.
- Failure Specification: case-specific expected-versus-actual behavior.
- Reproduction Bundle: immutable redacted inputs, fixtures, assumptions, and hashes.
- Repair Task: leased bounded handoff to one Repair Worker attempt.
- Repair Candidate: immutable proposed target revision and verification plan.
- Verification Receipt: independent signed targeted verification result.

### Extension Contracts

- Workflow Runtime Adapter describes workflow/revision identity, receives or retrieves execution observations, requests supported replay, and reports health.
- Repair Delivery Adapter maps generic repair operations into target-native candidate, review, promotion, and rollback behavior.
- Repair Delivery Binding selects exact adapter, target, credential reference, allowed operations, and policies without containing secrets.
- Operational Packages distribute adapter implementations, schemas, mappings, fingerprint rules, redaction defaults, extraction rules, health checks, compatibility, and tests.

The first n8n Package exports both Runtime and Repair Delivery Adapters. New integrations require no Kernel semantics.

## Trust And Custody

- Raw diagnostics and Diagnostic Worker execution remain customer-controlled.
- One PostgreSQL instance uses separate Kernel and Diagnostic databases and roles.
- Content-addressed local storage holds snapshots, payloads, bundles, patches, and verification artifacts.
- Payload retention may delete bytes while preserving digest tombstones.
- n8n provider credentials remain in n8n.
- Repair runtime credentials remain in the attached customer worker.
- Alphonse stores neither credential class.
- Console/API bind to loopback by default.
- V0.2 has one bootstrap Owner Principal and short-lived machine identities.
- Repair generation, verification, and promotion use separate identities and authority.

## Lifecycles

Diagnostic Cases, Repair Tasks, Repair Candidates, and Promotions use separate append-only state machines. Retries create new attempts, candidates remain immutable, failed work remains visible, and target promotion creates its own idempotent receipt. Projections may summarize but cannot flatten these truths.

The affected agent may report failure but cannot adjudicate itself. A separate Diagnostic Worker creates diagnosis artifacts. A customer-selected Repair Worker such as Codex proposes changes. A deterministic Verification Runner evaluates the exact candidate. A human initially retains promotion authority.

## Delivery Sequence

1. Headless deterministic proof through CLI and HTTP.
2. Model-assisted Diagnostic Worker using the same confirmed artifact boundary.
3. Five-view Builder Console over proven operations: Workflows, Execution, Diagnostic Case, Repair, Promotion.
4. OTLP plus TypeScript Agent SDK as the second-runtime portability proof.
5. Broader runtime adapters, governed customer-agent effects, Butler, and optional ALPHONSE_DATA based on observed demand.

## V0.2 Acceptance

The architecture proof must complete from fresh local state using public documentation, preserve exact revisions and receipts, fail the original fixture, pass the candidate, promote only with human authority, preserve rollback and regression artifacts, and remain idempotent under repeated events and commands.

It must not use direct n8n database access, store provider credentials in Alphonse, send a real customer email, require broad agent evaluation, require OTLP or Agent SDK, or place n8n-specific semantics inside Kernel.

## Deferred

- Automatic anomaly detection
- Broad agent evaluation or certification
- Automatic merge, promotion, or production deployment
- OTLP collector and general Agent SDK before the n8n proof
- Managed hosting and shared multi-tenancy
- Teams, SSO, and broad RBAC
- Butler operator product integration
- Required ALPHONSE_DATA adoption
- Public untrusted executable adapter packages
- Marketplace mechanics
- Kubernetes and distributed infrastructure

## Expansion Rule

Expand only when real workflows expose repeated need. The first proof validates the generic contracts; the second runtime validates portability; unfamiliar builders validate usability. Kernel V0.1 remains stable unless the proof demonstrates a required contract correction.
