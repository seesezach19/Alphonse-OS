# Three-run diagnostic consistency

Ticket 17 registers one Diagnostic Consistency Test against the completed Ticket 16 Worker Run and
its exact frozen Evidence Package. Registration commits a structured hidden rubric to content-addressed
storage and immutable database facts before creating exactly three fresh, unclaimed Assignments.
Registration grants no dispatch, model-request, repair, or external-effect authority.

Each Assignment still requires its own short-lived Kernel Dispatch Authorization, atomic Diagnostic
Plane claim, isolated Worker launch, and single-use Model Broker Grant. Before a consistency Worker
can launch, the Diagnostic Plane canonicalizes the declared runtime image and isolation policy, model
and sampling controls, Broker policy, resource limits, data and egress policies, exact package,
instruction, output schema, capabilities, and prohibitions. Ephemeral Assignment, Worker Run, launch,
nonce, and timestamp identities are excluded. The first digest becomes the test binding; either later
run fails closed before Model Broker contact if its digest differs.

The Worker output contract carries a neutral closed taxonomy for mechanism, observed and required
identity scope, support, confidence, implementation-location status, delivery/operation cardinality,
typed package citations, recommended investigations, and an exact empty `actions_taken` array. It
contains no rubric default, constant, example answer, or consistency-test identifier. Completion
resolves all five required citation roles against the exact package and scores each immutable diagnosis
independently against the precommitted rubric. It does not classify free prose with keyword or regular
expression rules and never creates a consensus diagnosis.

After all three runs, one immutable report separates:

- platform reproducibility: three distinct Assignments and Worker Runs launched with one exact
  configuration digest;
- model consistency: three-of-three structured rubric results plus confidence variance, pairwise
  evidence overlap, structured unsupported-claim count, recommendation convergence, and prose
  divergence.

All three diagnosis rows and their individual score rows remain independently addressable, even when
identical diagnosis content shares one content digest.

## Assurance limits

- The canonical proof uses a deterministic synthetic reference provider. It proves boundary,
  configuration, rubric-commitment, ingestion, and scoring behavior—not frontier-model quality or
  statistical reliability.
- The reference Broker can assert a model label but cannot provider-verify a hosted model snapshot or
  seed. Both limitations are recorded in every run configuration and the final report.
- Deterministic software scores only the closed structured fields. It reports prose divergence but does
  not infer semantic equivalence or unsupported claims from free prose.
- A passing result remains authority-free: it establishes neither external failure truth nor permission
  to repair or produce a business effect.
