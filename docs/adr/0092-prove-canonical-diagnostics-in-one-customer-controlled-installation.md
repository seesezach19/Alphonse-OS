# Prove Canonical Diagnostics In One Customer-Controlled Installation

The first vertical proof implements one installation and environment in Docker Compose with logically separated
Postgres authority and Diagnostic Plane state, local atomic filesystem CAS and garbage collection, private networks,
unique HMAC key, Principal, Reporting Grant and stream per observer, one Diagnostic API, durable Stage Worker,
scheduler, separate Kernel authority service, dispatcher, isolated runner, narrow Model Broker, n8n, public local
ingress adapter, separate Tokenization Service, and mock CRM with append-only commit ledger.

The proof ships only source, runtime, request, effect-feed, and snapshot observers; exact pre-execution n8n binding;
five immutable first-party observation schemas; one Integration Behavior Contract and Behavior Contract; one
correlation projector, effect interpreter, count-by-correlation evaluator, deterministic collection and selection
policy, evidence package, and assignment contract. Installation, environment, binding, schema, contract, grant, and
stream identities remain explicit even in the single-customer deployment.

Hosted tenancy, remote observers, public evidence gateways, mTLS, asymmetric signing, adapter marketplace, batching,
SDK and MCP surfaces, generalized rules, arbitrary package code, additional providers, cryptographic host attestation,
and distributed CAS are deferred. Completion requires deterministic stimulus-only assignment creation, one governed
isolated worker, and a preregistered three-run consistency test. Controller-authored evidence, raw source identities
or keys, nondurable stages, raw-effect evaluation, mutable packaging, assignment-implied authority, or instruction-only
worker isolation fail the proof.
