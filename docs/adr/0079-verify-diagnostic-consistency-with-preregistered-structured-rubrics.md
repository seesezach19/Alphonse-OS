# Verify Diagnostic Consistency With Preregistered Structured Rubrics

A deployed Diagnostic Consistency Test Policy may create three independent assignments over one exact evidence
package despite the normal one-assignment default. Every assignment binds the same instructions, output schema,
model claim and settings, limits, image, isolation and mount policies, broker policy, and tool availability. A
common Worker Run Configuration Digest excludes only assignment, run, worker, ephemeral credential, and timestamp
identities. Unverifiable provider snapshot or sampling controls limit the reproducibility claim explicitly.

Before dispatch, the verifier commits a hidden structured rubric, verifier artifact and version, expected package
digest, and rubric digest. Diagnostic output includes structured causal fields rather than relying on keyword or
regex interpretation of prose. Each run must independently identify delivery-scoped versus logical-operation-scoped
idempotency, distinguish attempts from operations, cite required material, resolve all citations, avoid unsupported
implementation claims, preserve assignment provenance, and have dispatcher-enforced proof of zero forbidden effects.

Three of three runs must pass. Platform reproducibility and model consistency are reported separately. All diagnoses
remain immutable individual artifacts; no vote or synthesized consensus replaces them. This is a strong first
repeatability proof, not a broad statistical reliability claim.
